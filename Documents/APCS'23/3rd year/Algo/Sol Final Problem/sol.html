Foundational building blocks

  • O(log min(a,b)) Euclidean GCD After two steps the smaller argument shrinks by at least half (Fibonacci worst case). Steps ≤ 2 log₂(min(a,b)).
  • O(log n) Binary search Each comparison halves the live interval. n → n/2 → … → 1 takes ⌈log₂ n⌉ steps.
  • O(log n) Heap push / pop A binary heap has height ⌊log₂ n⌋. Each push/pop bubbles up or down at most that many levels.
  • O(n log n) Merge sort / comparison sort Recurrence T(n)=2T(n/2)+O(n); Master Theorem case 2 → O(n log n). Also the information-theoretic lower bound: log₂(n!) ≈ n log n.
  • O(n) Prefix / suffix arrays One left-to-right and one right-to-left pass; each element touched once per pass.
  • O(N log log N) Sieve of Eratosthenes Total markings = Σ_{p prime ≤ N} N/p = N·Σ 1/p. By Mertens' theorem Σ 1/p ≈ log log N.
  • O(√S) Sum of two squares search Try a from 0 to √S; for each check if S−a² is a perfect square. At most √S iterations.

Strings, sequences & divide-and-conquer

  • O(|A|+|B|) KMP string matching Counter k increments at most |A| times; each fall-back decreases k. Total fall-backs ≤ increments ≤ |A|. π build costs O(|B|) identically.
  • O(n log n) Merge-sort inversion count Same recurrence as merge sort: T(n)=2T(n/2)+O(n). During merge, when a[i]>a[j], all (mid−i+1) remaining left elements form inversions — counted in O(1) per step.
  • O(N log k) k-way merge (min-heap) Each of the N total elements is pushed and popped from the heap exactly once. Each operation costs O(log k) since heap size ≤ k.
  • O(|s|·|p|) Regex DP (Problem 9) (|s|+1)×(|p|+1) table; each cell O(1) from two previous cells.
  • O(n·k·log V) Sliding-window GCD n windows × k GCD calls × O(log V) per call.

Dynamic programming

  • O(m·n) 2-D grid DP (Problem 10) Fill each of the m×n cells in O(1) using three neighbours. Two passes (one per value). Rolling row cuts space to O(n).
  • O(s·p) 2-D DP table (Problem 9) Two nested loops, each cell O(1). Space reducible to O(p) with rolling array.
  • O(k·n·m) 3-D DP table (Problem 11) Three nested loops, each cell O(1) from three candidates. Space O(n·m) with rolling i-slice.
  • O(n log V) Binary search on answer (Problem 19) O(log V) binary search iterations × O(n) cost evaluation per iteration.
  • O(n²) Permutation rank (Problem 20) For each of n positions, scan remaining elements to find the digit: O(n) per position, O(n²) total. A Fenwick tree reduces to O(n log n).
  • O(n²) Polygon diagonal (Problem 14) O(n) outer × O(n) inner loop with O(1) incremental area update per step.

Graph, tree & combinatorics

  • O(M log M) Kruskal's MST Sort M edges: O(M log M). Union-find operations: O(M α(N)) ≈ O(M).
  • O(M log M + K·M) K-th MST MST once + K heap extractions each generating ≤ M candidate trees.
  • O(N+M) Tarjan's bridge finding One DFS: each vertex and edge visited exactly once. disc[] and low[] updated in O(1) per step.
  • O(n) Tree width via edge contributions One DFS for subtree sizes (O(n)), then sum sz[c]×(n−sz[c]) over n−1 edges (O(n)).
  • O(n² log n) Angular sweep (Problem 13) n pivots × O(n log n) sort + O(n) sweep each.
  • O(n) BFS on remainders (Problem 17) n states, each enqueued once, 2 transitions each.
  • O(1) Circle union area (Problem 15) Fixed arithmetic: one sqrt, one arccos, one sin — no loops.

Search & enumeration

  • O(n!) Backtracking upper bound (Problem 8) Without pruning: n choices for column 1, n−1 for column 2, … → n! paths. Conflict pruning eliminates most branches early — actual solutions for n=9 is 352 out of 362880.
  • O(n · n!) BFS on permutation states (Problem 18) State space = n! permutations. Each state has n−1 transitions, each costing O(n) to compute a prefix reversal. For n=8: 8×40320 ≈ 322K ops.
  • O(n²) Goldbach counting (Problem 2) Outer loop over i from 2 to n, inner loop over primes p up to 2i. Sieve is O(N log log N) but the counting dominates at O(n²).
  • O(N log k) k-way merge (again) Reminder: total elements N, heap size ≤ k, each element pushed/popped once.

Concept Lessons

Concept 1

GCD — Euclidean Algorithm

The Greatest Common Divisor (GCD) of two integers a and b is the largest integer that divides both. The Euclidean algorithm computes it by repeatedly replacing the larger number with the remainder.

How it works

gcd(a, 0) = a                        // base case: anything divides 0
gcd(a, b) = gcd(b, a mod b)          // recursive step

The key identity is: any common divisor of a and b also divides (a mod b), because a mod b = a − ⌊a/b⌋ · b — a linear combination of a and b. So the set of common divisors of (a, b) equals the set of common divisors of (b, a mod b), and in particular their GCDs are equal.

Worked example

gcd(48, 18)
→ gcd(18, 48 mod 18) = gcd(18, 12)
→ gcd(12, 18 mod 12) = gcd(12, 6)
→ gcd(6, 12 mod 6) = gcd(6, 0)
→ 6

Why O(log min(a, b))?

After every two steps, the smaller argument shrinks by at least half. Proof: if b ≤ a/2 then the new pair is (b, a mod b) and b ≤ a/2. If b > a/2 then a mod b = a − b < a/2, so the argument after two steps is less than a/2. The smaller number therefore halves every two recursive calls, giving at most 2 log₂(min(a,b)) steps total → O(log min(a,b)).

When to use

  • Computing GCD of a window or array (Problem 1)
  • Checking if two numbers are coprime (gcd = 1)
  • LCM: lcm(a,b) = a / gcd(a,b) * b (divide first to avoid overflow)

Concept 2

Binary Search — Finding a Value in a Sorted Array

Given a sorted array and a target value, binary search finds the target's position (or the insertion point) by repeatedly halving the live search interval.

How it works

lo = 0;  hi = n − 1
While lo ≤ hi:
    mid = (lo + hi) / 2
    If a[mid] == target:   return mid          // found
    Elif a[mid] < target:  lo = mid + 1       // target is in the right half
    Else:                   hi = mid − 1       // target is in the left half
Return −1   // not found

lower_bound (first index ≥ target) and upper_bound (first index > target) are variants that never check for exact equality — they just move lo or hi until lo == hi.

Worked example

Array: [1, 3, 5, 7, 9, 11], target = 7
lo=0, hi=5 → mid=2, a[2]=5 < 7 → lo=3
lo=3, hi=5 → mid=4, a[4]=9 > 7 → hi=3
lo=3, hi=3 → mid=3, a[3]=7 == 7 → return 3

Why O(log n)?

Each iteration cuts the search interval in half: n → n/2 → n/4 → … → 1. The number of halvings needed to reach 1 from n is exactly ⌈log₂ n⌉. So at most ⌈log₂ n⌉ + 1 comparisons are made → O(log n).

When to use

  • Finding closest element to a target in a sorted array (Problem 3)
  • Checking if a value exists in a large sorted dataset
  • Finding boundaries (first/last occurrence) — use lower_bound / upper_bound variants

Concept 3

Binary Search on the Answer (Parametric Search)

Instead of searching for a value in an array, we binary-search the answer space itself. This works whenever the feasibility condition is monotone: if answer h is feasible, then every answer h′ on one side of h is also feasible.

The pattern

lo = minimum_possible_answer
hi = maximum_possible_answer

While lo < hi:
    mid = (lo + hi) / 2
    If feasible(mid):
        hi = mid          // mid works, try for a better answer
    Else:
        lo = mid + 1      // mid doesn't work, need to relax

Output lo

The key step is designing feasible(mid) — a function that returns true/false and runs efficiently (typically O(n)).

Worked example — Problem 19 (minimum height)

We want the minimum height h such that COST(h) ≤ T.
Observation: if we can achieve height h with cost ≤ T, then we can definitely achieve height h+1 (just excavate less) with cost ≤ T too.
So feasibility is monotone increasing in h: there is a threshold h* where
  h < h*: too expensive (COST > T)
  h ≥ h*: feasible (COST ≤ T)
Binary search finds h* directly without trying every value.

Why this works

Monotonicity turns an optimisation problem ("find the best h") into a sequence of decision problems ("is h feasible?"). Binary search on [lo, hi] runs in O(log(hi−lo)) iterations, each calling feasible() once. Total cost: O(log(range) × cost_of_feasible). For Problem 19: O(log(max_a + T) × n) → O(n log V).

When to use

  • Any "minimise the maximum" or "maximise the minimum" problem
  • When directly computing the optimum is hard, but checking "is X achievable?" is easy
  • Common trigger phrase: "find the minimum X such that [condition on X]"

Concept 4

Sorting + Binary Search (or Two-Pointer)

Many problems on two sequences A and B reduce to: for each element of A, find the "best matching" element in B. Sorting B first enables O(log n) lookup per query instead of O(n).

Binary search variant (Problem 3)

Sort B.
For each a[i]:
    target = f(a[i])                    // what we want from B
    pos = lower_bound(B, target)        // O(log n)
    Check B[pos] and B[pos−1] (floor and ceiling of target).
    Update global best.

Two-pointer variant (for sum = 0 style)

Sort A ascending, Sort B descending.
i = 0 (left pointer on A),  j = 0 (left pointer on B)
While i < m and j < n:
    s = a[i] + b[j]
    Update best if |s| is smaller.
    If s < 0:  i++    // need a larger value from A
    Else:       j++   // need a smaller value from B (B is sorted desc so j++ means smaller)

Worked example — Problem 3

A=[1,8,2,9], B=[−5,−6,3,−7,−4]. Goal: minimise |a+b|, i.e. find b closest to −a.
Sort B: [−7,−6,−5,−4,3].
For a[2]=8, target=−8. lower_bound finds pos=0 (first ≥ −8). Ceiling = −7. |8+(−7)|=1 ← best.
Output: 2 4 (original index of −7 is 4)

Key insight

Sorting is a one-time O(n log n) investment that reduces every subsequent search from O(n) to O(log n). If there are m queries, total cost drops from O(m·n) to O(n log n + m log n) = O((m+n) log n).

Concept 5

KMP — Knuth-Morris-Pratt String Matching

KMP finds all occurrences of pattern B in text A in O(|A|+|B|) time by never re-examining a character of A. The key is the failure function π, which encodes how much of the pattern can be "recycled" after a mismatch.

Failure function π

π[i] = length of the longest proper prefix of B[0..i] that is also a suffix of B[0..i].

Example: B = "ababc"
  B[0]='a': π[0]=0  (no proper prefix)
  B[1]='b': π[1]=0  ("a" is not a suffix of "ab")
  B[2]='a': π[2]=1  ("a" is a prefix and suffix of "aba")
  B[3]='b': π[3]=2  ("ab" is a prefix and suffix of "abab")
  B[4]='c': π[4]=0  ("ababc" has no matching prefix-suffix)
  π = [0, 0, 1, 2, 0]

Using π during search

k = 0  // characters of B matched so far
For each character A[i]:
    While k > 0 and B[k] ≠ A[i]:
        k = π[k−1]      // fall back: reuse the longest prefix that still matches
    If B[k] == A[i]:  k++
    If k == |B|:
        record match at position  i − |B| + 2
        k = π[k−1]      // allow overlapping matches

Worked example

A = "aaaaa", B = "aa", π = [0,1].
i=0: A[0]='a'=B[0], k=1
i=1: A[1]='a'=B[1], k=2=|B| → match at pos 1; k=π[1]=1
i=2: A[2]='a'=B[1] (k=1), k=2 → match at pos 2; k=1
i=3: match at pos 3; i=4: match at pos 4.
Output: 1 2 3 4

Why O(|A| + |B|)?

Track k as a potential function. k can increase by at most 1 per character of A (total: ≤|A| increments). Each fall-back step (k = π[k−1]) strictly decreases k. Therefore total fall-backs ≤ total increments ≤ |A|. Both the build of π (same argument over B) and the search are linear → O(|A|+|B|).

Concept 6

Prefix and Suffix Arrays

A prefix array P stores a cumulative aggregate (sum, max, min, XOR, …) of a[1..i] for every i. A suffix array stores the same aggregated from the right. Once built in O(n), they answer range queries in O(1).

Prefix sum

P[0] = 0
P[i] = P[i−1] + a[i]          // build in O(n)

Sum(l, r) = P[r] − P[l−1]     // range sum in O(1)

Prefix max (used in rain water, Problem 7)

L[1] = a[1]
L[i] = max(L[i−1], a[i])      // left max up to i

R[n] = a[n]
R[i] = max(R[i+1], a[i])      // right max from i

Worked example — range sum

a = [3, 1, 4, 1, 5, 9]
P = [0, 3, 4, 8, 9, 14, 23]
Sum(2, 5) = P[5] − P[1] = 14 − 3 = 11 (1+4+1+5=11 ✓)
Without prefix sums this would require 4 additions; with prefix sums it's 1 subtraction.

Key insight

Prefix arrays trade O(n) preprocessing for O(1) per query. If you need to answer Q range queries, total cost drops from O(Q·n) to O(n + Q). The trick generalises to any associative operation that has an inverse (e.g. XOR, addition, multiplication with modular inverse). For operations without inverses (like max), you need a Sparse Table or Segment Tree instead.

The split-at-j trick (used in Problem 19)

cost(j, h) = Σᵢ max(0, a[i] − h − |i−j|)
           = Σᵢ≤j max(0, (a[i]+i) − (h+j))     ← left part
           + Σᵢ>j max(0, (a[i]−i) − (h−j))     ← right part

Precompute prefix sums of (a[i]+i) and (a[i]−i).
For each j: evaluate both parts in O(1) using the prefix arrays.
Sweep over all j → O(n) total instead of O(n²).

Concept 7

Dynamic Programming (1-D, 2-D, 3-D)

Dynamic programming (DP) solves problems with overlapping subproblems and optimal substructure by storing the result of each subproblem so it is computed only once. The solution is built bottom-up from smaller subproblems.

Two requirements

  • Optimal substructure: the optimal solution to the full problem can be built from optimal solutions to subproblems.
  • Overlapping subproblems: the same subproblem is encountered many times; memoising it avoids redundant work.

1-D DP — one state variable

dp[i] = answer considering the first i elements
dp[i] = best combination of dp[j] (j < i) + cost of extending from j to i

Example: longest increasing subsequence
dp[i] = max(dp[j] + 1) for all j < i with a[j] < a[i]

2-D DP — two state variables (Problem 9)

dp[i][j] = can s[1..i] be matched by p[1..j]?

Depends on dp[i−1][j−1], dp[i][j−2], dp[i−1][j].
Fill table row by row; each cell O(1).
Total: O(|s| × |p|).

3-D DP — three state variables (Problem 11)

dp[i][j][l] = max weight using first i clusters,
              j in Group 1, l in Group 2.

dp[i][j][l] = max(dp[i−1][j][l],          // skip
                  dp[i−1][j−1][l] + a[i], // G1
                  dp[i−1][j][l−1] + b[i]) // G2

Total: O(k × n × m) states, each O(1).

Space optimisation — rolling array

When dp[i] only depends on dp[i−1] (and not earlier rows), you only need to keep two rows in memory at once: the current and the previous. This reduces space from O(k·n·m) to O(n·m) for a 3-D DP, or from O(n·m) to O(m) for a 2-D DP. The trade-off is that you cannot reconstruct the solution without re-running or storing backpointers separately.

Order of computation matters

Always fill dp in an order where every cell's dependencies are already computed.
For dp[i][j] depending on dp[i−1][j−1]: fill i from 1→n and j from 1→m (or reverse for '*' case in regex DP).
Mislabelling or wrong iteration order gives wrong answers silently — always trace through a small example.

Concept 8

BFS on an Implicit State Graph

Many problems that don't look like graph problems can be modelled as one: define states as nodes and valid transitions as edges. BFS then finds the shortest path (fewest transitions) from a start state to a goal state in O(|states| + |transitions|).

Template

visited = empty set
queue = [(start_state)]
visited.add(start_state)

While queue not empty:
    state = queue.dequeue()
    If state is goal:  return steps_to_reach(state)
    For each next_state reachable from state:
        If next_state not in visited:
            visited.add(next_state)
            queue.enqueue(next_state)

Why BFS gives shortest path (fewest steps)

BFS explores states layer by layer: all states reachable in 1 step, then 2 steps, then 3, … The first time the goal is reached, it is via the minimum number of steps. This relies on all edges having equal cost (weight = 1). For weighted edges use Dijkstra's algorithm instead.

Worked example — Problem 17 (modular BFS)

States: remainders {0, 1, …, n−1}. Start: state 1 (digit "1"). Goal: state 0.
Transitions from state r: append digit 0 → new state (r·10+0) mod n;
                          append digit 1 → new state (r·10+1) mod n.
For n=6: 1→{4,5}→{2,3,…}→…→0 first reached via "1110".
BFS guarantees "1110" is the shortest (= smallest) such number.

State graph size determines complexity

  • Problem 17: n states (remainders 0..n−1), 2 transitions each → O(n)
  • Sliding puzzle (3×3): 9!/2 ≈ 181,000 states
  • Always ask: "How many distinct states can there be?" — that is your worst-case complexity.

Concept 9

Computational Geometry — Polar Angles and Circular Segments

Two geometric primitives used in these problems: (a) polar angle to sort points around a pivot, and (b) the circular segment formula to compute intersection areas.

Polar angle and atan2

angle(P, Q) = atan2(Q.y − P.y, Q.x − P.x)
            = angle in (−π, π] of the vector from P to Q

Use atan2(dy, dx) (not atan(dy/dx)) because atan2 correctly handles all four quadrants and the case dx = 0.

Cross product — which side of a line?

cross(A→B, A→C) = (B.x−A.x)·(C.y−A.y) − (B.y−A.y)·(C.x−A.x)
  > 0  →  C is strictly to the LEFT  of directed line A→B
  = 0  →  C is collinear with A and B
  < 0  →  C is strictly to the RIGHT of directed line A→B

This is the primary tool for all "which side of a line" questions in Problem 13.

Circular segment area — Problem 15

Two equal circles, radius R, centres distance d apart.

If d ≥ 2R:  intersection = 0
Else:
    α = 2·arccos(d / (2R))          // full angle of the chord at each centre
    segment = R²·(α − sin α) / 2   // one circular segment (sector − triangle)
    intersection = 2·segment
    union = 2πR² − intersection

Worked example — segment formula

R=2, d=2√2. α = 2·arccos(√2/2) = 2·(π/4) = π/2.
Sector area = R²·α/2 = 4·(π/2)/2 = π.
Triangle area = R²·sin(α)/2 = 4·sin(π/2)/2 = 2.
Segment = π − 2 ≈ 1.142. Intersection = 2·1.142 = 2.283.
Union = 8π − 2.283 ≈ 22.850.

Key insight

The segment formula comes from: Sector (pie slice) − Triangle (the pointy bit inside the chord). The sector has angle α and area R²α/2. The triangle has base = chord length and height = distance from centre to chord = R·cos(α/2); its area simplifies to R²·sin(α)/2. Subtracting gives R²(α − sin α)/2.

Concept 10

Sieve of Eratosthenes

An algorithm to find all prime numbers up to N by iteratively marking the multiples of each prime as composite. Used in Problem 2 (even-numbered) to count prime pairs quickly.

Algorithm

is_prime[0..N] = all true
is_prime[0] = is_prime[1] = false

For p = 2 to √N:
    If is_prime[p]:
        For multiple = p² to N step p:
            is_prime[multiple] = false

// All indices where is_prime[i] = true are prime.

We start marking at p² (not 2p) because all smaller multiples of p have already been marked by earlier primes.

Worked example — primes up to 20

p=2: mark 4,6,8,10,12,14,16,18,20
p=3: mark 9,15 (6,12,18 already marked; start at 9=3²)
p=4: not prime, skip. √20 ≈ 4.47, done.
Primes: 2, 3, 5, 7, 11, 13, 17, 19

Why O(N log log N)?

The total work is proportional to the number of markings: Σ_{p prime ≤ N} N/p. By Mertens' second theorem, the sum of reciprocals of all primes up to N converges to log log N as N → ∞. So total markings ≈ N · log log N → O(N log log N). In practice log log N grows extremely slowly (log log 10⁶ ≈ 2.9), making the sieve essentially linear for any practical N.

Concept 11

Minimum Spanning Trees — Kruskal's Algorithm

A spanning tree of a connected graph is a subgraph that includes all vertices and is a tree (connected, no cycles). The Minimum Spanning Tree (MST) is the spanning tree with the smallest total edge weight. Kruskal's algorithm builds the MST greedily.

Algorithm

Sort all M edges by weight (ascending).
Initialise Union-Find with N isolated nodes.

MST_weight = 0;   edges_used = 0
For each edge (u, v, w) in sorted order:
    If find(u) ≠ find(v):         // u and v are in different components
        union(u, v)               // merge their components
        MST_weight += w
        edges_used++
    If edges_used == N−1:  break  // MST is complete (N−1 edges)

Worked example

Edges sorted: (2,5,1),(1,2,2),(2,4,3),(1,3,4),(3,5,5)
Add (2,5,1): {2,5} ∪ {1} ∪ {3} ∪ {4} — OK, weight=1
Add (1,2,2): {1,2,5} ∪ {3} ∪ {4} — OK, weight=3
Add (2,4,3): {1,2,4,5} ∪ {3} — OK, weight=6
Add (1,3,4): {1,2,3,4,5} — OK, weight=10. Done (4 edges = N−1).
MST weight = 10

Why greedy works — the Cut Property

Cut property: For any partition of the vertices into two non-empty sets S and V\S, the minimum-weight edge crossing the cut belongs to some MST. Proof: suppose edge e (the lightest crossing edge) is not in MST T. Adding e to T creates a cycle; that cycle must cross the cut at least twice, so there's another crossing edge e′ in T. Since w(e) ≤ w(e′), swapping e′ for e gives a spanning tree of equal or lesser weight that includes e → e is in some MST. Kruskal's processes edges in weight order, always picking the safest crossing edge → O(M log M).

Concept 12

Union-Find (Disjoint Set Union — DSU)

Union-Find maintains a collection of disjoint sets and supports two operations efficiently: find (which set does x belong to?) and union (merge the sets containing x and y). It is the backbone of Kruskal's cycle detection.

Data structure

parent[i] = i         // initially each node is its own root
rank[i]   = 0         // tree height estimate

find(x):
    If parent[x] ≠ x:
        parent[x] = find(parent[x])   // path compression: flatten the tree
    Return parent[x]

union(x, y):
    rx = find(x);  ry = find(y)
    If rx == ry: return             // already in same set
    If rank[rx] < rank[ry]:  swap(rx, ry)
    parent[ry] = rx                 // union by rank: attach smaller tree under larger
    If rank[rx] == rank[ry]:  rank[rx]++

Worked example

5 nodes, add edges one by one for Kruskal:
union(2,5): parent[5]=2, rank[2]=1
union(1,2): find(1)=1, find(2)=2 → parent[1]=2, rank unchanged
union(2,4): find(2)=2, find(4)=4 → parent[4]=2
union(1,3): find(1)=find(parent[1])=find(2)=2; find(3)=3 → parent[3]=2
Now find(1) = find(3) = find(4) = find(5) = 2 — all in one set.

Why nearly O(1) per operation?

Path compression (during find) flattens every node directly under the root. Union by rank keeps trees shallow. Together they give amortised O(α(n)) per operation, where α is the inverse Ackermann function — α(n) ≤ 4 for any n that could exist in the physical universe. For all practical purposes this is O(1). Without these two optimisations, find costs O(n) in the worst case.

Concept 13

Matroid Theory — Why K-th MST Works

A matroid is an abstract structure that generalises "independence" from linear algebra to combinatorics. Spanning trees of a graph form a graphic matroid, and this structure is exactly why we can enumerate spanning trees by single edge-swaps.

The exchange property

In any matroid: if I₁ and I₂ are two independent sets (e.g. two spanning trees) and |I₁| = |I₂|, then for any element e ∈ I₁ \ I₂ there exists an element f ∈ I₂ \ I₁ such that (I₁ − {e} ∪ {f}) is still independent.

For spanning trees: if T₁ and T₂ are two spanning trees, you can always swap one edge of T₁ for one edge of T₂ and still have a valid spanning tree. This is not obvious — it requires proof that the graph remains connected and acyclic.

Consequence for K-th MST

Any spanning tree T' can be reached from T₀ (the MST) by:
    1. Add one non-tree edge e  (creates a cycle C)
    2. Remove one tree edge eₜ from C  (restores tree property)
    Repeat until T' is reached.

The weight change per swap: Δw = w(e) − w(eₜ)
  If w(e) > w(eₜ): the tree gets heavier (moving away from MST)
  If w(e) < w(eₜ): impossible since eₜ was chosen for the MST (cut property)
  So Δw ≥ 0: all spanning trees are at least as heavy as the MST.

Why the priority queue gives trees in order

Every spanning tree T′ other than T₀ has at least one "first swap" — the single lightest-weight swap that produces it from some predecessor. By always extracting the minimum from the priority queue (min-heap), and generating all 1-swap neighbours of each extracted tree, we enumerate spanning trees in non-decreasing order of total weight. The K-th extraction is therefore the K-th MST. The canonical ordering (only consider swaps "forward" in the edge list) ensures no tree is generated twice.

Concept 14

Monotone Cost Functions

A function f(x) is monotone non-decreasing if x₁ ≤ x₂ implies f(x₁) ≤ f(x₂). Recognising that your cost (or feasibility) function is monotone is what unlocks binary search on the answer.

How to recognise monotonicity

  • Ask: "If I relax the target (make the constraint weaker), does the cost go down or stay the same?" If yes → monotone.
  • Ask: "Can I prove by a direct argument that a feasible solution at level x gives a feasible solution at level x+1?" If yes → monotone.

Problem 19 — COST(h) is monotone in h

COST(h) = min cost to make some position reach height h.
If h increases by 1 (shallower target): every position needs to be reduced by 1 less unit, so cost decreases or stays equal.
If h decreases by 1 (deeper target): every position must be reduced 1 more unit, so cost increases or stays equal.
∴ COST(h) is non-increasing in h → there is a threshold h* where COST(h) crosses budget T. Binary search finds h*.

General template

// Minimise h subject to COST(h) ≤ T, where COST is non-increasing
lo = hardest_possible_h
hi = easiest_possible_h
While lo < hi:
    mid = (lo + hi) / 2
    If COST(mid) ≤ T:  hi = mid    // mid is feasible, try harder
    Else:              lo = mid+1  // mid is infeasible, need to relax

Key insight

Without monotonicity, you might need to try every possible answer. With it, binary search reduces the search from O(range) to O(log range). The power is that even when COST(h) takes O(n) to evaluate, binary search brings the total to O(n log range) — a dramatic improvement over the naive O(n · range).

Concept 15

Prefix-Sum Trick for Range Queries

Build a prefix-sum array P in O(n) so that any range sum (or other aggregate) can be answered in O(1). The trick extends to 2-D, weighted sums, and conditional sums.

1-D prefix sum

P[0] = 0
P[i] = P[i−1] + a[i]           // O(n) to build

Sum(l, r) = P[r] − P[l−1]      // O(1) per query

Count of elements satisfying a condition

// Count elements in a[l..r] with value > threshold T:
C[i] = C[i−1] + (a[i] > T ? 1 : 0)
Count(l, r) = C[r] − C[l−1]    // O(1)

Application in Problem 19 — evaluating all j in O(n)

cost(j, h) = Σᵢ≤j max(0, a[i] − h − (j − i))
           = Σᵢ≤j max(0, (a[i]+i) − (h+j))

Let v[i] = a[i] + i.
For a fixed j, only indices where v[i] > h+j contribute:
    contribution = Σ_{v[i]>h+j, i≤j} (v[i] − (h+j))
                 = (sum of v[i] for contributing i) − (h+j)·(count of contributing i)

Build prefix-sum of v[i] sorted by value → binary search for threshold h+j
→ evaluate left part in O(log n). Similarly for the right part.
Sweep j from 1 to n → O(n log n) total (or O(n) with a two-pointer sweep).

Key insight

The prefix-sum trick is applicable whenever: (1) you need to compute the same aggregate over many different ranges, (2) the aggregate is a linear function (sum, count, weighted sum). It converts a naive O(n) per-query cost to O(1) per query at the cost of O(n) preprocessing. In 2-D, the same idea gives O(1) rectangular-region queries after O(nm) preprocessing.

Concept 16

Divide and Conquer

Divide the problem into smaller independent subproblems, solve each recursively, then combine the results. The key is that the combination step is cheap relative to the subproblems — typically O(n) — which drives the total cost to O(n log n).

The pattern

solve(problem of size n):
    If n is small:  solve directly (base case)
    left  = solve(left half,  size n/2)
    right = solve(right half, size n/2)
    return combine(left, right)   // O(n) work

Master Theorem for T(n) = aT(n/b) + O(nᶜ): if a = bᶜ (case 2) → T(n) = O(n log n). Merge sort has a=2, b=2, c=1 so a=b^c → O(n log n).

Application — counting inversions (Problem 4)

An inversion is a pair (i,j) with i<j and a[i]>a[j].
Three mutually exclusive categories:
    (A) Both i,j in left half   → count recursively
    (B) Both i,j in right half  → count recursively
    (C) i in left, j in right   → count during merge

Counting type (C) during merge:
    Both halves are already sorted.
    When a[i] > a[j] during merge:
        All remaining left elements a[i..mid] also beat a[j]
        → add (mid − i + 1) inversions in O(1)

Worked example

a = [3, 1, 2]. Split: [3] and [1,2].
Left inversions = 0. Right inversions: merge [1] and [2] → 0.
Merge [3] with [1,2]: a[i]=3 > a[j]=1 → add (mid−i+1)=1. Output 1; then 2; then 3.
Cross inversions = 2. Total = 0+0+2 = 2. (Pairs: (3,1) and (3,2).) ✓

Why O(n log n)?

T(n) = 2T(n/2) + O(n). The recursion tree has log₂ n levels; at each level the total work across all sub-problems at that level is O(n). So total = O(n) × log n levels = O(n log n). This is also the lower bound for comparison-based sorting, so merge sort is optimal.

Concept 17

Min-Heap / Priority Queue

A min-heap is a complete binary tree where every node's value is ≤ its children. It supports extracting the minimum element and inserting new elements, both in O(log n). A priority queue is the abstract data type backed by a heap.

Structure and key operations

Stored as an array: children of node i are at 2i and 2i+1; parent at ⌊i/2⌋.
Heap property: heap[i] ≤ heap[2i] and heap[i] ≤ heap[2i+1]

push(x):                         // O(log n)
    Append x at the end (index n+1).
    Bubble UP: while x < parent, swap with parent.

pop_min():                       // O(log n)
    Save heap[1] (the minimum).
    Move last element to position 1; shrink size.
    Bubble DOWN: while x > a child, swap with the smaller child.

peek_min():  return heap[1]      // O(1)

Why O(log n) per operation?

A complete binary tree of n nodes has height ⌊log₂ n⌋. Bubble-up and bubble-down each traverse at most one root-to-leaf path → at most ⌊log₂ n⌋ comparisons and swaps → O(log n). Building a heap from n elements naively is O(n log n) but the heapify-all-from-bottom trick gives O(n) — useful when you know all elements upfront.

Application — k-way merge (Problem 6)

Maintain one heap entry per non-exhausted sequence. Heap size ≤ k at all times.
Pop minimum (O(log k)), output it, push next element from its sequence (O(log k)).
N total elements → N pushes and N pops → O(N log k) total.
This beats the naive O(N·k) sequential scan approach for large k.

When to use

  • Repeatedly extracting the minimum (or maximum) from a changing set
  • Dijkstra's shortest path, Prim's MST, Huffman coding
  • K-th MST (Problem 21): candidate spanning trees sorted by weight
  • Any "always process the cheapest next" strategy

Concept 18

Backtracking

Backtracking is systematic trial-and-error: build a solution incrementally, and whenever a partial solution violates a constraint, undo the last choice ("backtrack") and try the next alternative. It is exhaustive but pruning makes it far faster than brute force.

Template

solve(depth, partial_solution):
    If partial_solution is complete:
        record it;  return

    For each choice c available at this depth:
        If choice c is VALID (doesn't violate constraints):
            Apply c to partial_solution
            solve(depth + 1, partial_solution)
            Undo c from partial_solution    // backtrack

Application — N-Queens (Problem 8)

Place queens column by column (depth = column index).
Choice at column j: which row r to place the queen.
Constraint: no two queens share a row, main diagonal, or anti-diagonal.
Check in O(1) using three boolean arrays:
    row_used[r]          // row r occupied?
    diag1_used[r − j]   // main diagonal r−j occupied?
    diag2_used[r + j]   // anti-diagonal r+j occupied?

Apply:   set all three to true.
Undo:    set all three to false.

Why O(1) conflict checking matters

Without the boolean arrays, checking validity at column j requires scanning all j previously placed queens — O(j) per row attempt → O(n²) per complete path. With the three arrays, each check is O(1) regardless of j. This pruning-plus-O(1)-check combination is what keeps N-Queens tractable for n ≤ 9 (352 solutions out of a theoretical 9! = 362880 paths).

Worked example (n=4)

j=0: try r=0 → place Q, mark row 0, diag1[−0=0], diag2[0+0=0].
j=1: r=0 blocked (row); r=1 blocked (diag2[1+1=2? No. diag1[1−1=0] ← blocked!); r=2: row OK, diag1[2−1=1] OK, diag2[2+1=3] OK → place Q.
j=2: all rows blocked → backtrack. Try r=3 at j=1: diag1[3−1=2] OK, diag2[3+1=4] OK → place Q.
j=2: r=1: diag1[1−2=−1] OK, diag2[1+2=3] blocked (diag2[4] was set? No — diag2[3+1=4]) ... continue until two solutions found.

Concept 19

2-D Grid DP — Largest All-Same Square

A classic grid DP pattern: dp[i][j] = side length of the largest square of value v with its bottom-right corner at cell (i,j). The recurrence links a cell to its three neighbours (top, left, top-left) and runs in O(mn).

Recurrence

If grid[i][j] ≠ v:
    dp[i][j] = 0                  // cell doesn't match; no square here

Else:
    dp[i][j] = min( dp[i−1][j],
                    dp[i][j−1],
                    dp[i−1][j−1] ) + 1

Why the min of three neighbours?

Suppose dp[i−1][j]=3, dp[i][j−1]=3, dp[i−1][j−1]=2. The top-left neighbour limits us to a 2×2 square in the overlap — even though the top and left each support 3×3, the top-left corner only supports 2×2 there. A square of side k at (i,j) requires all-v squares of side k−1 at all three neighbours simultaneously. The bottleneck is the smallest of the three. The +1 extends that square by including the current cell.

Worked example

Grid (4×5, only 1s shown for v=1):
Row 0: 1 0 1 0 0
Row 1: 1 0 1 1 1
Row 2: 1 1 1 1 1
Row 3: 1 0 0 1 0

dp for v=1 at (2,2): min(dp[1][2]=1, dp[2][1]=1, dp[1][1]=0)+1 = 0+1 = 1
dp at (2,3): min(dp[1][3]=1, dp[2][2]=1, dp[1][2]=1)+1 = 1+1 = 2
dp at (2,4): min(dp[1][4]=1, dp[2][3]=2, dp[1][3]=1)+1 = 1+1 = 2 ← max = 2

When to use

  • Largest rectangle of 1s (extend to use a histogram-stack approach for O(mn))
  • Any "maximum square/rectangle satisfying a condition" problem on a 2-D grid
  • Run twice (for v=0 and v=1) and take the overall max

Concept 20

Shoelace Formula — Polygon Area

The shoelace formula computes the exact area of any simple (non-self-intersecting) polygon given its vertices in order, using only integer arithmetic (if coordinates are integers). It runs in O(n).

Formula

Vertices (x₁,y₁), (x₂,y₂), …, (xₙ,yₙ) listed in counter-clockwise order.

2A = |Σᵢ₌₁ⁿ (xᵢ·yᵢ₊₁ − xᵢ₊₁·yᵢ)|      (indices mod n; i.e. xₙ₊₁ = x₁)

Area = 2A / 2

Multiply by 2 first and divide at the end to keep arithmetic in integers (since each cross-product term xᵢyᵢ₊₁ − xᵢ₊₁yᵢ is an integer). This avoids floating-point errors entirely.

Geometric intuition — why it works

Each term (xᵢyᵢ₊₁ − xᵢ₊₁yᵢ) is the signed area of the trapezoid formed by edge i and the x-axis (or equivalently, twice the signed area of the triangle (origin, vᵢ, vᵢ₊₁)). Signed areas add for counter-clockwise winding and cancel for regions covered twice, leaving exactly the polygon's area. The absolute value handles clockwise ordering.

Cross product connection

cross(A, B, C) = (B.x−A.x)·(C.y−A.y) − (B.y−A.y)·(C.x−A.x)
              = 2 × area of triangle ABC   (signed)

Area of polygon = (1/2)|Σ cross(origin, vᵢ, vᵢ₊₁)|
               = shoelace formula with origin = (0,0)

Application — Problem 14 (incremental fan area)

For diagonal (i,j) in the polygon, area of sub-polygon (i, i+1, …, j) is:
2A = |cross(vᵢ,vᵢ₊₁,−) + … + cross(vⱼ₋₁,vⱼ,−) + cross(vⱼ,vᵢ,−)|
Accumulate the cross products incrementally as j grows → each new j adds one term in O(1).
This avoids recomputing from scratch for every diagonal.

Concept 21

Tree DP — Subtree Sizes and Edge Contributions

Many tree problems are solved by rooting the tree, computing a value bottom-up for each subtree (DFS post-order), then answering queries using those values. The "edge contribution" pattern converts a sum-of-all-pairs problem into a sum over edges.

Computing subtree sizes

Root tree at vertex 1. DFS post-order:

DFS(v, parent):
    sz[v] = 1
    For each neighbour u ≠ parent:
        DFS(u, v)
        sz[v] += sz[u]          // accumulate children

After DFS: sz[v] = total number of vertices in the subtree rooted at v (including v).

Edge contribution formula — Problem 16

For any pair (u, v), their path uses a set of tree edges.
For a single edge e = (parent, child):
    e is on the path (u,v) ⟺ u and v are on OPPOSITE sides of e
    ⟺ exactly one of u,v is in the subtree of child.

Number of such pairs = sz[child] × (n − sz[child])

Total sum of all pairwise distances = Σ_{each edge e} sz[child(e)] × (n − sz[child(e)])

Worked example

Tree: 1−2, 1−3, 3−4, 3−5. n=5. Root at 1.
sz: leaf 2→1, leaf 4→1, leaf 5→1, node 3→3, root 1→5.
Edge 1-2: 1×4=4. Edge 1-3: 3×2=6. Edge 3-4: 1×4=4. Edge 3-5: 1×4=4.
Total = 4+6+4+4 = 18 ✓

Verify by brute force: d(1,2)=1, d(1,3)=1, d(1,4)=2, d(1,5)=2, d(2,3)=2, d(2,4)=3, d(2,5)=3, d(3,4)=1, d(3,5)=1, d(4,5)=2. Sum=18 ✓

Why O(n) vs O(n²)?

The brute-force approach runs BFS/DFS from each of n vertices to find all distances: O(n²). The edge-contribution formula replaces this with one DFS (O(n)) and one pass over n−1 edges (O(n)) — total O(n). The key insight is that we never need to know which specific pairs cross an edge, only how many.

Concept 22

Factorial Number System — Permutation Rank

The factorial number system represents a non-negative integer as a sum of factorials with digits dₙ·n! + dₙ₋₁·(n−1)! + … + d₁·1! + d₀·0!, where 0 ≤ dᵢ ≤ i. It directly encodes the rank of a permutation in lexicographic order.

Encoding a permutation as its rank

available = [1, 2, …, n]  (sorted)
rank = 1

For position i from 1 to n:
    dᵢ = index of p[i] in available   // how many remaining elements are smaller
    rank += dᵢ × (n − i)!
    remove p[i] from available

dᵢ is the "digit" at position i: it counts how many permutations share the same prefix p[1..i−1] but have a smaller value at position i. Each such block has size (n−i)!.

Decoding a rank back to a permutation

available = [1, 2, …, n]
remainder = y − 1             // convert to 0-indexed

For position i from 1 to n:
    dᵢ       = remainder / (n − i)!     // integer division
    remainder = remainder mod (n − i)!
    q[i]     = available[dᵢ]            // pick the (dᵢ+1)-th smallest remaining
    remove q[i] from available

Worked example

n=4, p=(2,1,4,3). available=[1,2,3,4].
i=1: p[1]=2, d=1 (one smaller: 1). rank += 1×3!=6. available=[1,3,4].
i=2: p[2]=1, d=0. rank += 0×2!=0. available=[3,4].
i=3: p[3]=4, d=1 (one smaller: 3). rank += 1×1!=1. available=[3].
i=4: p[4]=3, d=0. rank += 0×0!=0.
rank = 1+6+0+1+0 = 8.

Reverse: y=8, remainder=7. (n=4)
i=1: d=7/6=1, rem=1. q[1]=available[1]=2. available=[1,3,4].
i=2: d=1/2=0, rem=1. q[2]=available[0]=1. available=[3,4].
i=3: d=1/1=1, rem=0. q[3]=available[1]=4. available=[3].
i=4: d=0. q[4]=3. → permutation (2,1,4,3) ✓

Key insight

The n! permutations are partitioned into n blocks of size (n−1)! by their first element. Within each block, permutations are sub-partitioned by their second element into blocks of (n−2)!. And so on. This is precisely the factorial number system. The rank is computed in O(n) digit-extractions; finding the available element at index dᵢ costs O(n) with a plain list (O(log n) with a Fenwick tree or order-statistics tree).

Concept 23

Tarjan's Bridge-Finding Algorithm

A bridge is an edge whose removal disconnects the graph. Tarjan's algorithm finds all bridges in O(N+M) using a single DFS by tracking two values per vertex: the discovery time disc[v] and the low-link value low[v].

Key definitions

  • disc[v] = the DFS discovery timestamp of v (0, 1, 2, …)
  • low[v] = minimum disc value reachable from the subtree rooted at v using tree edges and at most one back edge going upward

Bridge condition

Edge (parent → child) is a BRIDGE  ⟺  low[child] > disc[parent]

Meaning: no vertex in child's subtree has a back edge reaching
         any ancestor of parent — the only connection between
         child's subtree and the rest is the edge (parent, child).

DFS with low-link computation

timer = 0

DFS(v, par):
    disc[v] = low[v] = timer++
    sz[v] = 1

    For each neighbour u of v:
        If disc[u] == −1:           // unvisited → tree edge
            DFS(u, v)
            sz[v] += sz[u]
            low[v] = min(low[v], low[u])   // propagate low upward
            If low[u] > disc[v]:
                // (v, u) is a BRIDGE; subtree size = sz[u]
                pairs_disconnected = sz[u] × (N − sz[u])
        Elif u ≠ par:               // already visited → back edge
            low[v] = min(low[v], disc[u])  // back edge raises no further

Worked example

Graph: 1-2-3-4, with extra edge 5-1 and 3-5 (from Problem 22).
DFS from 1: disc/low after DFS:
  v=1: disc=0; v=2: disc=1; v=3: disc=2; v=4: disc=3, low=3 (leaf).
  Back at v=3: low[4]=3 > disc[3]=2 → edge 3-4 is a bridge. sz[4]=1, pairs=1×4=4.
  v=5: disc=4, back edge to 1 → low[5]=min(4, disc[1])=min(4,0)=0.
  Back at v=3: low[5]=0 ≤ disc[3]=2 → edge 3-5 is NOT a bridge.
  low[3]=min(2,low[4],low[5])=0 ≤ disc[2]=1 → edge 2-3 not a bridge. Etc.
Only bridge: (3,4). Output for each edge: 0 0 4 0 0 ✓

Why O(N+M)?

One DFS traversal visits every vertex once (O(N)) and processes every edge twice — once in each direction (O(M)). disc[] and low[] are updated in O(1) per vertex/edge. The bridge condition low[u] > disc[v] is checked in O(1) per tree edge. Total: O(N+M).

Concept 24

Sum of Two Squares — Fermat's Theorem

Fermat's theorem on sums of two squares characterises which positive integers can be written as a²+b² for integers a,b. This determines when Problem 12 (integer-area square construction) is possible.

The theorem

A positive integer S can be expressed as a sum of two integer squares if and only if in the prime factorisation of S, every prime of the form p ≡ 3 (mod 4) appears to an even power.

Prime classification

  • p = 2: 2 = 1²+1². Special case, always expressible.
  • p ≡ 1 (mod 4): e.g. 5,13,17,29,37,… These primes are always sums of two squares (e.g. 5=1²+2²). Any power of such a prime is also expressible.
  • p ≡ 3 (mod 4): e.g. 3,7,11,19,23,… These primes are never sums of two squares. If p³ divides S but p⁴ does not (odd power), then S cannot be expressed as a sum of two squares.

Quick check algorithm

For each prime p ≡ 3 (mod 4) up to √S:
    count = 0
    While S mod p == 0:
        S /= p;   count++
    If count is odd:  return "Impossible"
If remaining S > 1 and S ≡ 3 (mod 4):
    return "Impossible"    // S itself is a prime ≡ 3 mod 4
return "Possible"

Examples

S=5: 5≡1(mod 4) → possible. 5=1²+2² ✓
S=3: 3≡3(mod 4), power=1 (odd) → Impossible
S=9: 9=3². Power of 3 is 2 (even) → possible. 9=0²+3² ✓
S=12: 12=4×3. Power of 3 is 1 (odd) → Impossible
S=50: 50=2×5². Power of 5 is 2 (even), 2 is fine → possible. 50=1²+7²=5²+5² ✓
S=21: 21=3×7. Both 3 and 7 ≡ 3(mod 4), both to power 1 (odd) → Impossible

Geometric interpretation

A square with integer coordinates and one side being vector (a,b) has area = a²+b² (by the Pythagorean theorem applied to the side vector). The integer-coordinate constraint forces the area to be a sum of two integer squares. Fermat's theorem is exactly the criterion for when such a decomposition exists. When it does, a brute-force search over a from 0 to √S finds (a,b) in O(√S) steps.

Problem 1 — Maximum GCD of k Consecutive Elements

Given a sequence of n integers a₁,a₂,…,aₙ, find a contiguous subarray of exactly k consecutive elements whose GCD is maximised. (2 ≤ k ≤ n ≤ 10³, 1 ≤ aᵢ ≤ 10⁶)

1. Define Subproblem / State

Let:

  • g[i] = GCD of the window of k elements starting at index i:
    g[i] = gcd(a[i], a[i+1], …, a[i+k−1])
  • best = maximum value of g[i] over all valid starting indices i ∈ [1, n−k+1]

2. Recurrence

Build g[i] by accumulating GCDs across the window, then update best:

best = 0
For i = 1 to n−k+1:
    curr = a[i]
    For j = i+1 to i+k−1:
        curr = gcd(curr, a[j])       // extend window by one element
    best = max(best, curr)

GCD uses the Euclidean algorithm: gcd(a, b) = gcd(b, a mod b), base case gcd(a, 0) = a.

3. Base Cases

  • g[i] initialised to a[i] — a window of size 1
  • best = 0 before any window is evaluated

4. Proof of Correctness + Result + Complexity

Every contiguous window of exactly k elements is evaluated (i ranges over all valid starting positions). The Euclidean GCD is exact and associative, so the order of accumulation does not affect the result. Taking the maximum over all windows is exhaustive — the optimal window cannot be skipped.
Instance: n=10, k=3, a=[2,6,4,3,18,12,24,8,7,5]
Window i=5: gcd(3,18)=3, gcd(3,12)=3 → g[5]=3
Window i=6: gcd(18,12)=6, gcd(6,24)=6 → g[6]=6 ← maximum
Output: 6
Result: best
Complexity: O(n · k · log max_a) — n windows, each k−1 GCD calls, each O(log max_a). With n, k ≤ 10³ and max_a ≤ 10⁶ this is ~2×10⁷ operations. Space: O(1).

Problem 3 — Minimise |aᵢ + bⱼ|

Given sequences A (m elements) and B (n elements), find indices i, j minimising |aᵢ + bⱼ|. (m, n ≤ 10⁵, |aᵢ|, |bⱼ| < 2×10⁹)

1. Define Subproblem / State

Let:

  • B_sorted = array B sorted in non-decreasing order, each element paired with its original 1-based index
  • best(i) = the original index j in B minimising |a[i] + b[j]| for a fixed i
  • (ans_i, ans_j) = the pair achieving the global minimum over all i

Key observation: |aᵢ + bⱼ| = |bⱼ − (−aᵢ)|. For a fixed i this is minimised by the element of B numerically closest to the target t = −a[i]. After sorting B, that element is either the floor (largest b ≤ t) or the ceiling (smallest b ≥ t) — checking both covers all cases.

2. Algorithm

Sort B_sorted = [(b[1],1), …, (b[n],n)] by value.
global_min = ∞;   ans_i = ans_j = −1

For each i from 1 to m:
    target = −a[i]
    pos = lower_bound(B_sorted, target)   // first index with value ≥ target

    For candidate in {B_sorted[pos], B_sorted[pos−1]}:   // ceiling and floor
        val = |a[i] + candidate.value|
        If val < global_min:
            global_min = val
            ans_i = i
            ans_j  = candidate.original_index

Output ans_i, ans_j

3. Base Cases

  • global_min = ∞ before any pair is evaluated
  • Boundary check: if pos = 0 only the ceiling candidate exists; if pos = n only the floor candidate exists

4. Proof of Correctness + Result + Complexity

For a fixed aᵢ, the function |bⱼ − t| (t = −aᵢ) is convex in bⱼ. In the sorted array the minimum is achieved at the two neighbours of t — the floor and ceiling. Checking both guarantees the closest element is found. Iterating over all i ensures the global minimum is found.
Instance: m=4, n=5, a=[1,8,2,9], b=[−5,−6,3,−7,−4]
B_sorted = [(−7,4),(−6,2),(−5,1),(−4,5),(3,3)]
For i=2, a[2]=8, target=−8. lower_bound→pos=0, ceiling=(−7, idx=4).
|8+(−7)| = 1 ← minimum over all pairs.
Output: 2 4
Result: (ans_i, ans_j)
Complexity: Sort B: O(n log n). Per i: O(log n) binary search. Total: O((m+n) log n). Space: O(n).

Problem 5 — All Occurrences of Pattern B in String A (KMP)

Find all 1-indexed positions where string B appears in string A. (|A|, |B| ≤ 10⁶)

1. Define Subproblem / State

Let:

  • π[i] = length of the longest proper prefix of B[0..i] that is also a suffix of B[0..i] (the "failure function")
  • k = number of characters of B currently matched during the scan of A
  • A match is recorded whenever k == |B|, at 1-indexed start position i − |B| + 2

2. Algorithm

// Phase 1: Build failure function π for B
π[0] = 0;   k = 0
For i from 1 to |B|−1:
    While k > 0 and B[k] ≠ B[i]:   k = π[k−1]   // fall back
    If B[k] == B[i]:                k++
    π[i] = k

// Phase 2: Scan A using π
k = 0
For i from 0 to |A|−1:
    While k > 0 and B[k] ≠ A[i]:   k = π[k−1]   // mismatch: fall back
    If B[k] == A[i]:                k++
    If k == |B|:
        Record position  i − |B| + 2             // 1-indexed match start
        k = π[k−1]                               // allow overlapping matches

3. Base Cases

  • π[0] = 0 — a single character has no proper prefix/suffix
  • k = 0 at the start of both phases — no characters matched yet

4. Proof of Correctness + Result + Complexity

π[i] encodes the maximum safe fall-back: when a mismatch occurs at position k of B after matching B[0..k−1] against A[i−k..i−1], we know B[0..π[k−1]−1] = A[i−π[k−1]..i−1] (by the prefix-suffix property of π), so matching can resume from k = π[k−1] without revisiting characters of A. No position of A is ever revisited, and no match is skipped because every fall-back strictly decreases k and terminates. After a full match we fall back to π[|B|−1] to find overlapping occurrences.
Instance: A="aaaaa", B="aa"
Failure function: π=[0,1]
Scan: match ends at i=1→pos 1; fall back k=1; match ends at i=2→pos 2; … → positions 1 2 3 4
Output: 1 2 3 4
Result: all match start positions in increasing order
Complexity: Build π: O(|B|). Scan A: O(|A|). Total: O(|A|+|B|). Space: O(|B|) for π.

Problem 7 — Trapping Rain Water

Given terrain heights h[1..n] with both ends bordering the sea at level 0, compute total trapped water. (n ≤ 3×10⁴, hᵢ ≤ 10⁵)

1. Define Subproblem / State

Let:

  • L[i] = max height in h[1..i] — the tallest wall to the left of i (inclusive)
  • R[i] = max height in h[i..n] — the tallest wall to the right of i (inclusive)
  • water[i] = water trapped directly above position i = max(0, min(L[i], R[i]) − h[i])

The water level at position i is capped by the shorter surrounding wall. Any excess drains over that wall.

2. Recurrence

L[1] = h[1]
L[i] = max(L[i−1], h[i])        for i = 2..n

R[n] = h[n]
R[i] = max(R[i+1], h[i])        for i = n−1..1

water[i] = max(0, min(L[i], R[i]) − h[i])

total = Σᵢ water[i]

3. Base Cases

  • L[1] = h[1], R[n] = h[n]
  • At endpoints i=1 and i=n, one side is the sea (level 0), so water[1] and water[n] are always 0

4. Proof of Correctness + Result + Complexity

Water at position i sits at level min(L[i], R[i]) — it cannot exceed the shorter surrounding wall or it would spill. The terrain occupies h[i] units, so trapped water is the difference (or 0 if h[i] is taller). Since L and R are exact prefix/suffix maxima, each water[i] is computed correctly.
Instance: h=[0,1,0,2,1,0,1,3,2,1,2,1]
i=3 (h=0): L=1, R=3 → water=min(1,3)−0=1
i=5 (h=0): L=2, R=3 → water=2; i=6 (h=1): L=2, R=3 → water=1; (all others sum to 2)
Total = 1+0+2+1+1+0+0+0+0+1 = 6. Output: 6
Result: total
Complexity: Three linear passes (build L, build R, sum water). Total: O(n). Space: O(n) for L, R (reducible to O(1) with a two-pointer scan).

Problem 9 — Regex Matching with '.' and '*'

Given string s and pattern p where '.' matches any single character and 'c*' means zero-or-more 'c', determine if s is fully matched by p. (|s| ≤ 20, |p| ≤ 30)

1. Define Subproblem / State

Let:

  • dp[i][j] = true if the first i characters of s can be fully matched by the first j characters of p

Indices are 1-based; s[i] denotes the i-th character of s and p[j] the j-th character of p. match(c, x) = (c == x) or (x == '.').

2. DP Recurrence

Case A — p[j] is a normal character or '.':
    dp[i][j] = dp[i−1][j−1]  AND  match(s[i], p[j])

Case B — p[j] == '*'  (p[j−1] is the repeated character):
    // Zero occurrences of p[j−1]: treat "p[j−1]*" as empty, skip two pattern chars
    dp[i][j] = dp[i][j−2]
    // One or more occurrences: s[i] matches p[j−1], and s[1..i−1] already matched p[1..j]
    OR  ( dp[i−1][j]  AND  match(s[i], p[j−1]) )

3. Base Cases

  • dp[0][0] = true — empty string matches empty pattern
  • dp[i][0] = false for i > 0 — non-empty string cannot match empty pattern
  • dp[0][j]: true only if p[1..j] can match the empty string, i.e. it is a sequence of "x*" pairs. Recurrence: dp[0][j] = dp[0][j−2] if p[j]=='*', else false

4. Proof of Correctness + Result + Complexity

The recurrence covers every possible interpretation: a '*' either annihilates the preceding character (zero occurrences, skip back two positions in p) or extends a prior match by consuming one character of s (staying at the same j). The '.' wildcard is handled inside match(). By induction on i and j, dp[i][j] correctly reflects whether s[1..i] is matchable by p[1..j]. The final answer is dp[|s|][|p|].
Instance 1: s="aab", p="c*a*b"
dp[0][2]=true (c*→0 c's), dp[0][4]=true (a*→0 a's)
dp[1][4]=true (extend a* to match 'a'), dp[2][4]=true, dp[3][5]=true (match 'b')
Output: true

Instance 2: s="mississippi", p="mis*is*p*."
The final '.' requires exactly one more unmatched character, but 'i' at the end cannot be consumed → dp[11][10]=false.
Output: false
Result: dp[|s|][|p|] printed as "true" or "false"
Complexity: O(|s| × |p|) time and space. Space reducible to O(|p|) with a rolling array.

k clusters; cluster i yields weight aᵢ in Group 1 or bᵢ in Group 2. Select exactly n clusters into Group 1 and m into Group 2 (each cluster in at most one group). Maximise total weight. (k ≤ 10³, n+m ≤ k)

Let:

dp[i][j][l] = max of three choices for cluster i:
    (a) Skip cluster i:
            dp[i−1][j][l]
    (b) Assign cluster i to Group 1   (valid if j ≥ 1):
            dp[i−1][j−1][l] + a[i]
    (c) Assign cluster i to Group 2   (valid if l ≥ 1):

4. Proof of Correctness + Result + Complexity

Each cluster has exactly three mutually exclusive choices: skip, Group 1, or Group 2. The DP explores all combinations while enforcing the count constraints via indices j ≤ n and l ≤ m. Problem 13 — Line Splitting n Points into Two Equal Halves

Given n points (n even), no three collinear. Find two point indices whose connecting line splits the remaining n−2 points into exactly (n−2)/2 on each side. (n ≤ 10³)

1. Define Subproblem / State

Let:

  • half = (n−2)/2 — the required count on each side
  • For a fixed pivot P, sort all other n−1 points by polar angle around P
  • cnt(P, Q) = number of points strictly to the left of directed line P→Q

Key fact: as Q rotates around P, cnt(P,Q) changes by exactly ±1 at each step (no three points are collinear). So there always exists an angular position where cnt = half.

2. Algorithm (Angular Sweep per Pivot)

For each pivot i from 1 to n:
    Compute angle[j] = atan2(y[j]−y[i], x[j]−x[i])  for all j ≠ i
    Sort other points by angle in [0°, 360°)

    Initialize count = number of points in the open half-plane
                       in front of the direction to the first sorted point

    For each sorted point Q:
        If count == half:
            Output i and original_index(Q);  stop.
        Update count as the rotating half-plane advances past angle[Q]

3. Base Cases / Invariant

  • A valid pair always exists (guaranteed by the no-three-collinear condition and the argument below)
  • Initialise count by explicitly counting how many other points lie in the starting half-plane

4. Proof of Correctness + Result + Complexity

Since no three points are collinear, no point lies on line P→Q. As Q sweeps a full circle around P, cnt(P,Q) is an integer that increases and decreases by 1 at each sorted-angle step, taking every value from 0 to n−2. In particular cnt = (n−2)/2 is always achieved. The sorted angular sweep identifies the correct Q in O(n) after an O(n log n) sort. Trying every pivot P guarantees we find the answer.
Instance: n=6, points: (3,5),(1,3),(3,1),(6,1),(8,3),(6,5)
Pivot P=2=(1,3). At Q=5=(8,3): 2 points strictly above line 2→5, 2 points strictly below → half=2 satisfied.
Output: 2 5
Result: two point indices
Complexity: Per pivot: O(n log n) sort + O(n) sweep. Over all n pivots: O(n² log n). With n ≤ 10³ this is ~10⁷ operations.

Problem 15 — Area of Union of Two Equal-Radius Circles

Two circles, both radius R, centres (x₁,y₁) and (x₂,y₂). Compute area of their union to 3 decimal places. (All integer inputs ≤ 100)

1. Define Subproblem / State

Let:

  • d = Euclidean distance between centres: d = √((x₂−x₁)² + (y₂−y₁)²)
  • I = area of intersection of the two circles (the lens-shaped region)
  • By inclusion-exclusion: Union = |A| + |B| − I = 2πR² − I

Three geometric cases arise depending on d versus R:

2. Formula

d = sqrt((x2−x1)² + (y2−y1)²)

If d >= 2R:
    I = 0                          // disjoint (or externally tangent)

Elif d == 0:
    I = π × R²                    // identical circles

Else:
    // Half-angle α at each centre subtended by the chord of intersection:
    α = 2 × arccos(d / (2R))
    // Area of one circular segment (half the lens):
    segment = R² × (α − sin α) / 2
    I = 2 × segment

Union = 2 × π × R² − I
Output Union formatted to 3 decimal places.

3. Base Cases

  • d ≥ 2R: circles are disjoint or tangent, I = 0, union = 2πR²
  • d = 0: circles are identical, union = πR²
  • 0 < d < 2R: partial overlap, use the circular segment formula

4. Proof of Correctness + Result + Complexity

Inclusion-exclusion |A ∪ B| = |A| + |B| − |A ∩ B| is exact. The intersection of two equal circles is a symmetric lens formed by two identical circular segments. The half-angle α at each centre is arccos(d/2R), derived from the isosceles triangle formed by the two centres and one intersection point. The standard segment area formula R²(θ − sin θ)/2 is obtained by subtracting the triangle area from the sector area — it is exact.
Instance: (1,2),(3,4), R=2 → d=√8 = 2√2 ≈ 2.8284
α = 2·arccos(2.8284/4) = 2·arccos(0.7071) = 2·(π/4) = π/2
segment = 4·(π/2 − sin(π/2))/2 = 4·(π/2−1)/2 ≈ 1.1416
I = 2·1.1416 = 2.2832; Union = 8π − 2.2832 ≈ 25.1327 − 2.2832 = 22.850
Result: Union to 3 decimal places
Complexity: O(1) — closed-form arithmetic only.

Problem 17 — Smallest Multiple of n Using Only Digits {0, 1}

Find the smallest positive integer m with at most 9 digits, composed only of 0s and 1s, divisible by n. (n ≤ 100)

1. Define Subproblem / State

Let:

  • r = the current number's remainder modulo n
  • We build the number digit by digit, always starting with 1 (no leading zeros). Appending digit d updates the remainder: r_new = (r × 10 + d) mod n
  • BFS state = (remainder r, digit string constructed so far). We seek remainder 0

2. Algorithm (BFS on Remainder Graph)

Queue: enqueue (remainder=1, number_string="1")
Visited: boolean[n], all false;  Visited[1] = true

While queue not empty:
    (r, num) = dequeue
    If r == 0:  output num;  stop.

    For d in {0, 1}:
        r_new = (r × 10 + d) mod n
        If not Visited[r_new]:
            Visited[r_new] = true
            enqueue (r_new, num + str(d))

3. Base Cases

  • Initial enqueue: remainder = 1 mod n (the digit "1" alone), Visited[1] = true
  • Special case n=1: "1" itself is the answer (1 mod 1 = 0)
  • A solution always exists within 9 digits for n ≤ 100 (the search space 2⁹ = 512 vastly exceeds the n ≤ 100 distinct remainders)

4. Proof of Correctness + Result + Complexity

There are exactly n distinct remainders modulo n. BFS visits each remainder at most once (Visited array). BFS explores nodes in order of increasing string length, and left-to-right digit order within the same length — so the first time remainder 0 is reached, the path spells the lexicographically (and therefore numerically) smallest such number. The transition (r×10+d) mod n correctly models appending digit d.

BFS sequence: "1"→r=1; "10"→r=4; "11"→r=5; "100"→r=4(seen); "101"→r=5(seen); "110"→r=2; "111"→r=3; "1100"→r=2(seen); "1101"→r=3(seen); "1110"→r=0 ✓
Verify: 1110 / 6 = 185. Output: 1110

Problem 19 — Minimum Achievable Height with Budget T

n segments with heights hᵢ (consecutive heights differ by at most 1). Reducing any height by 1 costs 1 coin. Final heights must still satisfy |finalᵢ − finalᵢ₊₁| ≤ 1. Find the minimum achievable height using at most T coins. (n ≤ 10⁵, T ≤ 10¹⁸, hᵢ ≤ 10⁹)

1. Define Subproblem / State

Let:

  • h = candidate target minimum height (binary-searched; can be negative)
  • cost(j, h) = minimum coins to make position j reach height h while keeping |finalᵢ − finalᵢ₊₁| ≤ 1 everywhere
  • COST(h) = minⱼ cost(j, h) — the cheapest way to achieve minimum height h at any position

Key insight: if final[j] = h (deepest point), the constraint propagates outward — each step away from j can rise by at most 1, so final[i] ≤ h + |i − j|. Since we can only reduce heights (final[i] ≤ a[i]), the cost-minimising choice is:

final[i] = min(a[i], h + |i − j|)

So: cost(j, h) = Σᵢ max(0, a[i] − h − |i − j|)

2. Algorithm (Binary Search + Prefix-Sum Cost Evaluation)

Binary search on h:
    lo = −T  (cannot excavate deeper than T total coins allow)
    hi = max(a[i])  (no excavation: COST = 0)

While lo < hi:
    mid = (lo + hi) / 2
    If COST(mid) ≤ T:   hi = mid      // mid is achievable, try deeper
    Else:               lo = mid + 1  // too expensive, go shallower

Output lo.


Computing COST(h) in O(n) via prefix sums:
    cost(j, h) = Σᵢ≤j max(0, (a[i]+i) − (h+j))   ← left contribution
               + Σᵢ>j max(0, (a[i]−i) − (h−j))    ← right contribution

Precompute prefix sums of (a[i]+i) and (a[i]−i).
For each j in O(1), compute the two sums using the prefix arrays and
a pointer to the last index where (a[i]+i) > (h+j) and (a[i]−i) > (h−j).
Sweep j from 1 to n → COST(h) = min over all j, in O(n) total.

3. Base Cases / Monotonicity

  • COST(h) is non-decreasing as h decreases: deeper excavation is never cheaper
  • At h = max(aᵢ), COST = 0
  • Binary search is valid because COST is monotone in h

4. Proof of Correctness + Result + Complexity

The formula final[i] = min(a[i], h + |i−j|) is tight: any higher value violates the ≤1 constraint propagating from position j; any lower value wastes coins since we only need to reach height h at j. COST(h) is therefore the exact minimum coins needed to achieve minimum height h. Binary search correctly identifies the minimum h for which COST(h) ≤ T because of the monotone property.
Instance 1: n=4, T=3, a=[1,1,1,1]. Try h=−1, j=3 (1-indexed):
final = [min(1,0), min(1,−1), min(1,0), min(1,1)] = [0,−1,0,1]
cost = (1−0)+(1−(−1))+(1−0)+(1−1) = 1+2+1+0 = 4 > 3
Best j gives cost=3 ≤ T → h=−1 is achievable. COST(−2) > 3. Output: −1

Instance 2: n=4, T=3, a=[1,2,2,1]. h=0 at j=3: final=[1,1,0,1], cost=0+1+2+0=3 ≤ T. h=−1: cost > 3. Output: 0
Result: lo after binary search
Complexity: Binary search: O(log(max_a + T)) ≈ 60 iterations. Each cost evaluation: O(n). Total: O(n log(max_a + T)) ≈ 6×10⁶ operations. Space: O(n).

Problem 21 — K-th Minimum Spanning Tree

Given N cities, M roads (all lengths distinct), find the total length of the K-th minimum spanning tree. Two plans differ if at least one edge differs.

1. Define Subproblem / State

Let:

  • T₀ = the MST found by Kruskal's algorithm, weight W₀ (the 1st spanning tree)
  • A 1-swap neighbour of spanning tree T: add one non-tree edge e=(u,v,wₑ) — creating the unique cycle P — then remove one tree edge eₜ on P. New weight = W(T) + wₑ − w(eₜ)
  • PQ = min-priority queue of candidate spanning trees, ordered by total weight
  • Each PQ entry stores: (total weight, the swap pair (e, eₜ), pointer to the base tree)

2. Algorithm (Priority Queue + Edge Swap Enumeration)

Step 1 — Find T₀ via Kruskal's (sort edges, union-find):
    Sort M edges by weight. Add edges greedily if they don't form a cycle.

Step 2 — Seed the priority queue with 1-swap neighbours of T₀:
    For each non-tree edge e=(u,v,wₑ):
        Find path u→v in T₀ (unique tree path).
        For each tree edge eₜ on that path:
            Push (W₀ + wₑ − w(eₜ),  e, eₜ,  T₀)  into PQ.

Step 3 — Extract the K-th MST:
    count = 1     // T₀ is tree #1
    While count < K:
        (W, e, eₜ, T_base) = PQ.pop_min()
        count++
        If count == K:  output W;  stop.

        // Generate 1-swap neighbours of T = T_base + e − eₜ
        // Canonical ordering: only consider non-tree edges of T that come
        // strictly after e in the sorted edge list (avoids duplicates)
        T = T_base + {e} − {eₜ}
        For each non-tree edge e' of T with index > index(e):
            For each tree edge eₜ' on path u'→v' in T:
                Push (W + w(e') − w(eₜ'), e', eₜ', T) into PQ.

3. Base Cases

  • count = 1; T₀ is the 1st MST. If K=1 output W₀ immediately
  • Uniqueness of edge lengths means no two spanning trees share the same total weight, simplifying ordering
  • The canonical "forward-only" swap ordering ensures each spanning tree is enqueued exactly once

4. Proof of Correctness + Result + Complexity

By matroid theory, any spanning tree T' can be reached from T₀ by a sequence of single edge swaps. The priority queue always extracts the globally next-lightest spanning tree because: (1) every spanning tree T' that has not yet been extracted has at least one 1-swap neighbour of lighter weight reachable by the "lightest predecessor" in the sequence from T₀; and (2) the canonical forward ordering prevents the same tree from being generated more than once (each tree has a unique "first swap" that produced it). Therefore the K-th extraction gives exactly the K-th MST.
Instance: N=5, M=5, K=2. Edges: (1,2,2),(1,3,4),(2,4,3),(2,5,1),(3,5,5)
Kruskal MST T₀: {(2,5,1),(1,2,2),(2,4,3),(1,3,4)}, W₀=10.
Only non-tree edge: (3,5,5). Path 3→1→2→5 in T₀. Swap (1,3,4) out → W=10−4+5=11.
Next pop from PQ gives weight 11 → count=2=K. Output: 11
Result: total weight of the K-th spanning tree
Complexity: Kruskal: O(M log M). Per PQ extraction: O(M) to enumerate swaps. Total: O(M log M + K · M). Space: O(K · M) for the priority queue in the worst case.

Problem 2 — Cumulative Goldbach Pair Count

Let g(n) = number of pairs (p,q) with p+q=2n and p,q both prime (pairs differ if an element differs; count (p,q) and (q,p) as different when p≠q). Compute f(n) = g(2)+g(3)+…+g(n). (3 ≤ n ≤ 10³)

1. Define Subproblem / State

  • is_prime[x] = true if x is prime (built by Sieve up to 2n)
  • g[i] = number of ordered pairs (p,q) with p+q=2i, p,q prime
  • f[i] = f[i−1] + g[i] — the running cumulative sum

To count g[i]: iterate p from 2 to 2i−2 and check whether both p and 2i−p are prime. Since (p,q) and (q,p) are different when p≠q, we count all p in that range (each ordered pair is counted once).

2. Recurrence

Build Sieve of Eratosthenes up to 2n.

g[i] = 0
For p = 2 to 2i−2:
    If is_prime[p] AND is_prime[2i − p]:
        g[i]++

f[i] = f[i−1] + g[i]

3. Base Cases

  • f[1] = 0 (summation starts from i=2)

Problem 4 — Count Inversions

Count pairs (i,j) with i<j and a[i]>a[j] in a sequence of n integers. (n ≤ 5×10⁴, |aᵢ| ≤ 2³¹)

1. Define Subproblem / State

  • inv(l, r) = number of inversions within the subarray a[l..r]
  • Split into two halves; count inversions within each half recursively, plus cross inversions (pairs where i is in the left half and j is in the right half) counted during the merge step

2. Recurrence (Merge Sort)

merge_sort(a, l, r) → returns inversion count for a[l..r]:
    If l == r:  return 0

    mid = (l + r) / 2
    count  = merge_sort(a, l, mid)
    count += merge_sort(a, mid+1, r)
    count += merge_and_count(a, l, mid, r)   // cross inversions
    return count

merge_and_count(a, l, mid, r):
    i = l;  j = mid+1;  count = 0
    While i ≤ mid and j ≤ r:
        If a[i] ≤ a[j]:
            output a[i];  i++
        Else:
            // a[i] > a[j]: all remaining left elements a[i..mid] also invert with a[j]
            count += (mid − i + 1)
            output a[j];  j++
    Merge remaining elements.
    return count

3. Base Cases

  • inv(i, i) = 0 — a single element has no inversions
  • Cross inversions = 0 when one half is empty

4. Proof of Correctness + Result + Complexity

Every inversion (i,j) with i<j belongs to exactly one of three categories: both i,j in left half; both in right half; i in left, j in right. The first two are counted recursively. For cross inversions: during the merge, when a[i] > a[j], the left subarray is already sorted so a[i], a[i+1], …, a[mid] are all > a[j] — that's (mid−i+1) inversions in one step. No inversion is double-counted or missed.

Inversions: (3,4)→4>3, (3,6)→4>1, (4,6)→3>1, (2,6)→2>1, (1? No 1<1). Let's enumerate: (4,3),(4,1),(3,1),(2,1),(5,1) — wait: indices (i,j), i<j, a[i]>a[j]: (3,4): a[3]=4>a[4]=3 ✓; (3,6): a[3]=4>a[6]=1 ✓; (4,6): a[4]=3>a[6]=1 ✓; (2,6): a[2]=2>a[6]=1 ✓; (5,6): a[5]=5>a[6]=1 ✓. Total = 5.
Result: total inversion count
Complexity: T(n) = 2T(n/2) + O(n) → by Master Theorem: O(n log n). Space: O(n) auxiliary merge buffer.

Problem 6 — Merge k Sorted Sequences

1. Define Subproblem / State

  • Each sequence has a "current front element" — the smallest unprocessed element in that sequence
  • At each step, the globally smallest front element across all k sequences should be output next
  • Maintain a min-heap of size ≤ k holding one (value, seq_index, pos_in_seq) triple per non-exhausted sequence

2. Algorithm

Initialize min-heap H.
For each sequence i with at least one element:
    H.push( (seq[i][0], i, 0) )

While H is not empty:
    (val, i, pos) = H.pop_min()
    output val
    If pos + 1 < len(seq[i]):
        H.push( (seq[i][pos+1], i, pos+1) )

3. Base Cases

  • Empty sequences are never inserted into the heap
  • When a sequence is exhausted (pos+1 = len), no further push is made for it
  • Output is complete when the heap is empty (all sequences exhausted)

4. Proof of Correctness + Result + Complexity

Invariant: the heap always contains exactly one representative per non-exhausted sequence — the smallest remaining element in that sequence. Since each sequence is sorted, its representative is its global minimum. The heap's minimum across all representatives is therefore the globally smallest remaining element. Popping it and inserting the next from the same sequence maintains the invariant. By induction, every output element is the smallest remaining → the output is globally sorted.

Pop (2,3,0)→output 2, push (6,3,1). … continue → 1 1 2 3 4 4 5 6
Result: the merged sorted list

Problem 8 — N-Queens

Find all ways to place n queens on an n×n board such that no two queens share a row, column, or diagonal. Print the count, then all board configurations. (1 ≤ n ≤ 9)

1. Define Subproblem / State

  • Place queens column by column: in column j, choose which row r to place the queen
  • Track three conflict sets:
    • row_used[r] — row r already has a queen
    • diag1_used[r−j] — main diagonal (r−j = constant) already has a queen
    • diag2_used[r+j] — anti-diagonal (r+j = constant) already has a queen
  • placement[j] = the row chosen for the queen in column j

2. Algorithm (Backtracking)

solve(j):                          // place queen in column j
    If j == n:
        record current placement;  return
    For r = 0 to n−1:
        If row_used[r] or diag1_used[r−j] or diag2_used[r+j]:
            continue               // conflict, skip this row
        // Place queen
        placement[j] = r
        row_used[r] = diag1_used[r−j] = diag2_used[r+j] = true
        solve(j + 1)
        // Remove queen (backtrack)
        row_used[r] = diag1_used[r−j] = diag2_used[r+j] = false

Call solve(0).
Output count, then each placement as an n×n grid.

3. Base Cases

  • j == n: all n queens placed without conflict — record the solution
  • n == 1: exactly one solution — the single cell
  • Diagonal index r−j ranges from −(n−1) to n−1; offset by n−1 to use as array index

4. Proof of Correctness + Result + Complexity

Since exactly one queen is placed per column and we check row/diagonal conflicts before placing, every completed placement is valid by construction. Every valid placement is found because backtracking exhaustively tries all row choices for each column. The three conflict arrays provide O(1) conflict checking — no need to re-scan previous queens.

Solution 1: placement=[1,3,0,2] (0-indexed rows) → .Q.. / ...Q / Q... / ..Q.
Solution 2: placement=[2,0,3,1] → ..Q. / Q... / ...Q / .Q..
Output: m=2, then the two grids
Result: count m, then m grids
Complexity: Worst-case O(n!) placements explored (each column has n choices). With pruning, actual count is much smaller — e.g. n=9 has 352 solutions out of 9! = 362880 paths. Per solution: O(n) to print. Space: O(n) for the three conflict arrays.

Problem 10 — Largest Square of All-0s or All-1s

Given an m×n binary matrix, find the side length of the largest square submatrix containing only 0s or only 1s. (1 ≤ m,n ≤ 100)

1. Define Subproblem / State

  • dp[v][i][j] = side length of the largest all-v square whose bottom-right corner is at cell (i,j)
  • Run the DP independently for v=0 and v=1; answer = max over both

2. DP Recurrence

If grid[i][j] ≠ v:
    dp[v][i][j] = 0                        // cell doesn't match value v

Else (grid[i][j] == v):
    If i == 0 or j == 0:
        dp[v][i][j] = 1                    // border cell: square of size 1
    Else:
        dp[v][i][j] = min( dp[v][i−1][j],
                           dp[v][i][j−1],

3. Base Cases

  • First row (i=0) and first column (j=0): dp[v][i][j] = 1 if grid[i][j]==v, else 0
  • All dp values initialised to 0

4. Proof of Correctness + Result + Complexity

The recurrence is correct by the following argument: dp[v][i][j] ≥ k if and only if the three neighbours dp[v][i−1][j], dp[v][i][j−1], dp[v][i−1][j−1] are all ≥ k−1 (meaning there exist all-v squares of side k−1 ending at those three corners) AND grid[i][j]=v. Problem 16 — Width of a Tree (Sum of All Pairwise Distances)

  • For each edge e=(parent, child): removing e splits the tree into two parts of sizes sz[child] and n−sz[child]. Every pair with one vertex on each side contributes exactly 1 to this edge's usage count.

// DFS to compute subtree sizes

    DFS(c)
    sz[v] += sz[c]

width = 0

Problem 20 — Permutation Rank (Both Directions)

Given permutation (p₁,…,pₙ), find its lexicographic rank x. Given rank y, find the corresponding permutation (q₁,…,qₙ). (n ≤ 20)

1. Define Subproblem / State

The factorial number system: the rank of a permutation is determined position by position. At position i, the "digit" dᵢ = number of remaining unused values smaller than pᵢ. The rank (1-indexed) is:

x = 1 + d₁·(n−1)! + d₂·(n−2)! + … + dₙ·0!

The inverse (rank → permutation): decompose y−1 in the factorial number system to get d₁,d₂,…,dₙ, then reconstruct by picking the (dᵢ+1)-th remaining element at each position.

2. Algorithm

// Forward: permutation → rank
available = sorted list of all values [1..n]
x = 1
For i = 1 to n:
    d = index of p[i] in available   // 0-indexed count of smaller remaining values
    x += d × (n−i)!
    remove p[i] from available

// Backward: rank y → permutation
available = sorted list [1..n]
remainder = y − 1
For i = 1 to n:
    d = remainder / (n−i)!           // integer division
    remainder = remainder mod (n−i)!
    q[i] = available[d]              // (d+1)-th smallest remaining element (0-indexed)
    remove q[i] from available

3. Base Cases

  • Factorial: 0! = 1, 1! = 1, precompute fact[0..n]
  • The rank of the identity permutation (1,2,…,n) is 1 (all digits dᵢ = 0)
  • The rank of the reverse permutation (n,n−1,…,1) is n!

4. Proof of Correctness + Result + Complexity

The number of permutations of (n−i) elements is (n−i)!. The dᵢ-th digit (0-indexed) represents how many permutations of the remaining elements have a smaller first element — exactly dᵢ·(n−i)! permutations come before the current one in that "block". Summing over all positions gives the total number of permutations that precede p, hence rank x = 1 + Σ dᵢ·(n−i)!. The inverse is exact because each factorial digit is uniquely determined by integer division.
Forward: p=(2,1,3), n=3. available=[1,2,3].
i=1: p[1]=2, index=1 (one smaller: 1), x+=1×2!=2. available=[1,3].
i=2: p[2]=1, index=0, x+=0×1!=0. available=[3].
i=3: p[3]=3, index=0, x+=0×0!=0. x=1+2=3.

Backward: y=4, n=3. remainder=3. available=[1,2,3].
i=1: d=3/2=1, rem=1. q[1]=available[1]=2. available=[1,3].
i=2: d=1/1=1, rem=0. q[2]=available[1]=3. available=[1].
i=3: d=0/1=0. q[3]=available[0]=1.
Permutation: (2,3,1).

Problem 22 — Disconnected Vertex Pairs per Edge Removal

Given a connected undirected graph with N vertices and M edges, for each edge output the number of vertex pairs (u,v) that become disconnected when that edge is removed. (edges listed in input order)

1. Define Subproblem / State

  • An edge e is a bridge if removing it disconnects the graph. For a bridge splitting the graph into components of sizes s and N−s, the number of newly disconnected pairs = s × (N−s).
  • A non-bridge edge belongs to a cycle; the graph remains connected after its removal → 0 disconnected pairs.
  • Use Tarjan's DFS to classify every edge as bridge or non-bridge, and compute s for each bridge.

Let disc[v] = DFS discovery time of v, and low[v] = minimum disc value reachable from the subtree rooted at v using at most one back edge.

2. Algorithm (Tarjan's Bridge Finding)

timer = 0
disc[v] = low[v] = −1 for all v

DFS(v, parent):
    disc[v] = low[v] = timer++
    sz[v] = 1
    For each neighbour u of v:
        If disc[u] == −1:           // tree edge
            DFS(u, v)
            sz[v] += sz[u]
            low[v] = min(low[v], low[u])
            If low[u] > disc[v]:    // (v,u) is a bridge
                mark edge (v,u) as bridge with subtree_size = sz[u]
        Elif u ≠ parent:            // back edge (not the edge we came from)
            low[v] = min(low[v], disc[u])

Start DFS(1, −1).

For each edge e (in input order):
    If e is a bridge with subtree size s:
        output s × (N − s)
    Else:
        output 0

3. Base Cases

  • A tree with M = N−1 edges: every edge is a bridge
  • A cycle: no bridges (every edge is on the cycle)
  • low[v] > disc[parent] is the bridge condition: no back edge from v's subtree reaches above the parent, so removing (parent,v) disconnects the subtree
  • Multi-edges: if there are two edges between u and v, neither is a bridge — handle by tracking edge id instead of parent vertex

4. Proof of Correctness + Result + Complexity

low[v] is the earliest ancestor reachable from v's subtree. If low[v] > disc[parent(v)], then no vertex in v's subtree has a back edge to any ancestor of parent — the only connection between the subtree and the rest of the graph is the tree edge (parent,v). Removing it disconnects sz[v] vertices from N−sz[v] vertices, giving sz[v]×(N−sz[v]) disconnected pairs. Conversely, if low[v] ≤ disc[parent], there is a back edge providing an alternative path → not a bridge → 0 pairs.
Instance: N=5, M=5, edges: 1-2, 2-3, 3-4, 3-5, 5-1 (input order).
DFS tree (rooted at 1): 1→2→3→4; 3→5→1 (back edge).
low[4]=disc[4]=3; low[5]=low[1]=0; low[3]=min(disc[3],low[4],low[5])=0; low[2]=0; low[1]=0.
Bridge check: edge (3,4): low[4]=3 > disc[3]=2 → bridge. sz[4]=1. Pairs = 1×4 = 4.
All other edges: low ≤ disc of parent → not bridges → 0 pairs.
Output in input order: edge 1-2→0, 2-3→0, 3-4→4, 3-5→0, 5-1→0. Output: 0 / 0 / 4 / 0 / 0
Result: M lines, one integer per edge
Complexity: One DFS visiting each vertex and edge once: O(N + M). Space: O(N + M) for adjacency list, disc, low, sz arrays.
